def cosine_similarity(word_a, word_b, model):
    """Calculate cosine similarity with step-by-step breakdown."""
    a = model[word_a]
    b = model[word_b]

    magnitude_b = np.linalg.norm(b)
    similarity  = dot_product / (magnitude_a * magnitude_b)
    return similarity, dot_product, magnitude_a, magnitude_b

3.1 LRU Cache — What Is It and Why Should You Care?¶

The problem: Imagine you run a restaurant. A customer orders a dish. You cook it (slow, expensive). Five minutes later, another customer orders the exact same dish. Do you cook it again from scratch? No — you keep a few ready-to-serve portions in a warmer.

That's a cache — a fast-access storage for results you've already computed.

LRU = Least Recently Used. When the cache is full and you need to store something new, you throw out the item that hasn't been used in the longest time.

Why this matters in AI:

  • Every LLM API call costs money ($0.01-$0.10 per call)
  • If 100 users ask "What is RAG?" today, you only need to call the LLM once and cache the result
  • A good cache can cut API costs by 30-50%

Real-world analogy: Your browser cache. When you visit a website, images are cached locally so they load instantly next time instead of downloading again.

# Exercise 3.1: LRU Cache (Model Serving Context)
# Interview scenario: "Design a cache for model inference results.
# When the cache is full, evict the least recently used entry."

from collections import OrderedDict

class LRUCache:
    """Least Recently Used cache for model inference results.
    
    Real-world use: Cache LLM responses to avoid redundant API calls.
    Same query with same parameters → return cached result.
    """
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = OrderedDict()
        self.hits = 0
        self.misses = 0
    
    def get(self, key: str) -> str | None:
        if key in self.cache:
            self.cache.move_to_end(key)  # mark as recently used
            self.hits += 1
            return self.cache[key]
        self.misses += 1
        return None
    
    def put(self, key: str, value: str) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)  # evict oldest
    
    @property
    def hit_rate(self) -> float:
        total = self.hits + self.misses
        return self.hits / total if total > 0 else 0.0


# Simulate caching LLM responses
cache = LRUCache(capacity=3)

# User queries
queries = [
    ("What is RAG?", "RAG is Retrieval-Augmented Generation..."),
    ("Explain transformers", "Transformers use self-attention..."),
    ("What is RAG?", None),  # should hit cache
    ("Define embeddings", "Embeddings are dense vector representations..."),
    ("What is RLHF?", "RLHF is Reinforcement Learning from Human Feedback..."),
    ("What is RAG?", None),  # evicted — capacity was 3
]

for query, response in queries:
    cached = cache.get(query)
    if cached:
        print(f"  CACHE HIT:  '{query}' → '{cached[:40]}...'")
    else:
        if response:
            cache.put(query, response)
            print(f"  CACHE MISS: '{query}' → stored in cache")
        else:
            print(f"  CACHE MISS: '{query}' → was evicted (LRU)")

print(f"\nHit rate: {cache.hit_rate:.0%}")
print(f"Cache size: {len(cache.cache)}/{cache.capacity}")
print("\n--- In production, this saves $$$. Same query = no API call. ---")

3.2 Text Preprocessing — Cleaning Data Before AI Touches It¶

The problem: Real-world text is messy. It has HTML tags, weird Unicode characters, extra spaces, and inconsistent formatting. If you feed this mess directly to an LLM or embedding model, you get worse results.

Analogy: Before you cook, you wash and chop the vegetables. You don't throw a whole unwashed carrot into the pot. Text preprocessing is the washing and chopping.

What a preprocessing pipeline does:

  1. Strip HTML tags (<p>, <div>, etc.)
  2. Fix Unicode (smart quotes → regular quotes)
  3. Collapse extra whitespace
  4. Lowercase (optional — depends on use case)
  5. Truncate to fit model's token limit

Why "pipeline"? Each step feeds into the next, like an assembly line. The order matters — you strip HTML before normalizing whitespace.

# Exercise 3.2: Text Preprocessing Pipeline
# Interview scenario: "Write a preprocessing pipeline for text
# before it gets embedded or sent to an LLM."

import re
from typing import Callable

def normalize_whitespace(text: str) -> str:
    """Collapse multiple spaces/newlines into single space."""
    return re.sub(r'\s+', ' ', text).strip()

def remove_html_tags(text: str) -> str:
    """Strip HTML tags from text."""
    return re.sub(r'<[^>]+>', '', text)

def normalize_unicode(text: str) -> str:
    """Replace common unicode characters with ASCII equivalents."""
    replacements = {
        '\u2018': "'", '\u2019': "'",  # smart quotes
        '\u201c': '"', '\u201d': '"',
        '\u2014': '--', '\u2013': '-',  # em/en dash
        '\u2026': '...',                # ellipsis
    }
    for old, new in replacements.items():
        text = text.replace(old, new)
    return text

def truncate_to_tokens(text: str, max_tokens: int = 512) -> str:
    """Approximate token truncation (1 token ~ 4 chars)."""
    max_chars = max_tokens * 4
    if len(text) <= max_chars:
        return text
    return text[:max_chars].rsplit(' ', 1)[0] + '...'


class TextPipeline:
    """Composable text preprocessing pipeline."""
    def __init__(self, steps: list[Callable[[str], str]]):
        self.steps = steps
    
    def process(self, text: str) -> str:
        for step in self.steps:
            text = step(text)
        return text
    
    def process_batch(self, texts: list[str]) -> list[str]:
        return [self.process(t) for t in texts]


# Build pipeline
pipeline = TextPipeline([
    remove_html_tags,
    normalize_unicode,
    normalize_whitespace,
    str.lower,
    lambda t: truncate_to_tokens(t, max_tokens=128),
])

# Test with messy input
raw_texts = [
    '<p>The \u201cquick\u201d brown   fox\u2026  jumped over the <b>lazy</b> dog.</p>',
    '  Multiple    spaces   and\n\nnewlines\n  everywhere  ',
    '<div class="content">RAG pipelines use <em>semantic search</em> to find relevant docs.</div>',
]

3.3 Batch Embedding — Processing Documents at Scale¶

The problem: You have 10,000 documents to convert into embeddings (numerical vectors). You can't send them all at once — the API has limits. You can't send them one by one — it would take hours.

Analogy: Imagine doing laundry. You don't wash one sock at a time (too slow), and you can't fit your entire wardrobe in one load (too big). You do it in batches — one load at a time, tracking progress.

What this exercise builds:

  • Group documents into batches (typically 8-64 at a time)
  • Send each batch to the embedding API
  • Track progress so you know where you are
  • Normalize the results (make all vectors the same "length" for fair comparison)

Key term — L2 Normalization: Makes every vector length = 1.0. This is like converting all measurements to the same unit. Without it, a longer document might have a "bigger" vector that unfairly dominates similarity calculations.

import numpy as np
from sklearn.datasets import make_classification
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler

def sigmoid(z):
    """Sigmoid function: squashes any number into the range (0, 1).
    
    Think of it as a "confidence converter":
    - Input of 10  → output ~1.0 (very confident YES)
    - Input of -10 → output ~0.0 (very confident NO)
    - Input of 0   → output 0.5  (coin flip — no idea)
    """
    return 1 / (1 + np.exp(-np.clip(z, -500, 500)))

def binary_cross_entropy(y_true, y_pred):
    """Measures how wrong our predictions are.
    
    - If we predicted 0.99 and the answer was 1 → low loss (good!)
    - If we predicted 0.99 and the answer was 0 → high loss (bad!)
    - The model's goal: minimize this number.
    """
    eps = 1e-15  # tiny number to prevent log(0) which is undefined
    y_pred = np.clip(y_pred, eps, 1 - eps)
    return -np.mean(y_true * np.log(y_pred) + (1 - y_true) * np.log(1 - y_pred))

class LogisticRegressionScratch:
    def __init__(self, lr=0.1, epochs=100):
        self.lr = lr          # Learning rate: how big a step we take each update
        self.epochs = epochs  # How many times we loop through the entire dataset
        self.weights = None   # One weight per feature — learned during training
        self.bias = None      # Single bias term — like the y-intercept in y = mx + b
        self.losses = []      # Track loss over time (should decrease)
    
    def fit(self, X, y):
        """Train the model. This is where learning happens."""
        n_samples, n_features = X.shape
        self.weights = np.zeros(n_features)  # Start with all weights = 0
        self.bias = 0.0
        
        for epoch in range(self.epochs):
            # STEP 1: Forward pass — make predictions with current weights
            z = X @ self.weights + self.bias    # Linear combination (like y = mx + b)
            predictions = sigmoid(z)             # Squash to 0-1 range (probability)
            
            # STEP 2: Compute loss — how wrong are we?
            loss = binary_cross_entropy(y, predictions)
            self.losses.append(loss)
            
            # STEP 3: Compute gradients — which direction should we adjust?
            # This is calculus, but intuitively: "how much did each weight
            # contribute to the error?"
            error = predictions - y              # Positive = predicted too high
            dw = (X.T @ error) / n_samples       # Gradient for weights
            db = np.mean(error)                   # Gradient for bias
            
            # STEP 4: Update weights — take a small step in the right direction
            # This IS gradient descent. That's it. It's just subtraction.
            self.weights -= self.lr * dw
            self.bias -= self.lr * db
            
            if (epoch + 1) % 25 == 0:
                print(f"  Epoch {epoch+1:3d} | Loss: {loss:.4f}")
        
        return self
    
    def predict_proba(self, X):
        """Get probability scores (0.0 to 1.0)."""
        return sigmoid(X @ self.weights + self.bias)
    
    def predict(self, X, threshold=0.5):
        """Get final yes/no predictions. Above 0.5 = yes (1), below = no (0)."""
        return (self.predict_proba(X) >= threshold).astype(int)

4.2 PyTorch Training Loop — The Pattern Behind Every AI Model¶

What is PyTorch? A Python library for building and training neural networks. It's the most popular framework in AI research and increasingly in production.

What is a training loop? The repetitive process of:

  1. Forward pass — feed data through the model, get predictions
  2. Compute loss — measure how wrong the predictions are
  3. Backward pass — calculate which weights caused the errors (backpropagation)
  4. Update weights — nudge the weights to make fewer errors next time
  5. Repeat thousands of times

Analogy: Learning to throw darts.

  • You throw (forward pass)
  • You see how far from the bullseye you landed (loss)
  • You figure out what went wrong — too high? too far left? (backward pass)
  • You adjust your aim (update weights)
  • After 10,000 throws, you're consistently hitting near the center

Why this matters: This exact loop structure — forward, loss, backward, update — is used to train GPT, BERT, Stable Diffusion, and every other neural network. Learn it once, understand everything.

4.3 Cosine Similarity — How AI Measures "Sameness"¶

The problem: You have a user's question as a vector and 10,000 document vectors. How do you find the most relevant documents?

Cosine similarity measures the angle between two vectors. If they point in the same direction, they're similar (score close to 1.0). If they point in opposite directions, they're dissimilar (score close to -1.0).

Analogy: Imagine two people pointing at the sky.

  • If they both point at the same star → cosine similarity ≈ 1.0 (very similar)
  • If they point in completely different directions → cosine similarity ≈ 0.0
  • The length of their arms doesn't matter — only the direction they're pointing

Why cosine and not just distance? In high-dimensional space (768 dimensions for typical embeddings), Euclidean distance gets unreliable. Cosine similarity only cares about direction, which is more meaningful for text.

This is the core of: RAG retrieval, semantic search, recommendation systems, and duplicate detection.

import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """Cosine similarity between two vectors."""
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def search(query_embedding: np.ndarray, corpus_embeddings: np.ndarray,
           top_k: int = 3) -> list[tuple[int, float]]:
    """Find top-k most similar embeddings.
    Returns list of (index, similarity_score)."""
    # Vectorized cosine similarity against all docs
    norms = np.linalg.norm(corpus_embeddings, axis=1)
    query_norm = np.linalg.norm(query_embedding)
    similarities = corpus_embeddings @ query_embedding / (norms * query_norm)
    
    # Get top-k indices
    top_indices = np.argsort(similarities)[::-1][:top_k]
    return [(idx, similarities[idx]) for idx in top_indices]

4.4 K-Means Clustering — Grouping Things That Are Alike¶

The problem: You have 10,000 customer support tickets. You need to group them into categories — but you don't have labels. Nobody has manually tagged them.

K-Means automatically groups data points into K clusters based on similarity.

How it works (in plain English):

  1. Pick K random points as starting "cluster centers"
  2. Assign every data point to the nearest center
  3. Move each center to the middle of its assigned points
  4. Repeat steps 2-3 until nothing moves

Analogy: Imagine you're organizing a party into groups.

  • Place 4 team captains randomly in the room
  • Everyone walks to the nearest captain
  • Each captain moves to the center of their group
  • Everyone re-evaluates — the nearest captain might have changed
  • After a few rounds, the groups stabilize

AI use cases: Topic discovery in documents, customer segmentation, grouping similar user queries, anomaly detection (points far from any cluster center).

# Interview scenario: "Implement K-means. When would you use it in an AI system?"

import numpy as np
import matplotlib.pyplot as plt

def kmeans(X: np.ndarray, k: int, max_iters: int = 50) -> tuple:
    """K-means clustering from scratch.
    Returns (centroids, labels, inertia_history)."""
    n_samples = X.shape[0]
    
    # Initialize centroids randomly from data points
    rng = np.random.RandomState(42)
    centroid_indices = rng.choice(n_samples, k, replace=False)
    centroids = X[centroid_indices].copy()
    
    labels = np.zeros(n_samples, dtype=int)
    inertia_history = []
    
    for iteration in range(max_iters):
        # Assign each point to nearest centroid
        distances = np.array([
            np.linalg.norm(X - c, axis=1) for c in centroids
        ]).T  # shape: (n_samples, k)
        labels = np.argmin(distances, axis=1)
        
        # Track inertia (sum of squared distances to centroids)
        inertia = sum(
            np.sum((X[labels == i] - centroids[i]) ** 2)
            for i in range(k)
        )
        inertia_history.append(inertia)
        
        # Update centroids
        new_centroids = np.array([
            X[labels == i].mean(axis=0) if np.any(labels == i) else centroids[i]
            for i in range(k)
        ])
        
        # Check convergence
        if np.allclose(centroids, new_centroids):
            print(f"  Converged at iteration {iteration + 1}")
            break
        centroids = new_centroids
    
    return centroids, labels, inertia_history

4.5 Model Evaluation — Why "95% Accurate" Can Be Terrible¶

The problem: Your fraud detection model has 95% accuracy. Sounds great, right? But if only 5% of transactions are fraud, a model that says "not fraud" for everything also gets 95%. It catches zero fraud.

The key metrics:

Metric Question It Answers Analogy
Accuracy What % of all predictions were correct? Your overall test score
Precision When I said "fraud", how often was it actually fraud? Of everyone I accused, how many were guilty?
Recall Of all actual fraud, how much did I catch? Of all the criminals, how many did I catch?
F1 Score Balance between precision and recall A single "grade" that balances both

The tradeoff:

  • High precision, low recall = rarely raises alarms, but when it does, it's right (strict cop — few arrests, all guilty)
  • Low precision, high recall = catches everything, but many false alarms (paranoid cop — arrests everyone)
  • Which matters more? Depends on the business. Spam filter? High precision (don't hide real email). Cancer screening? High recall (don't miss a case).

Interview gold: "Which metric would you optimize for?" — The answer is always "it depends on the cost of false positives vs false negatives."

    fp = np.sum((y_pred == 1) & (y_true == 0))

    return {
        'accuracy': accuracy, 'precision': precision,
        'recall': recall, 'f1': f1,

    }

4.6 Attention Mechanism — How Transformers "Read"¶

The problem: When you read the sentence "The cat sat on the mat because it was tired" — you know "it" refers to "cat", not "mat". How does a model figure that out?

Attention lets each word look at every other word and decide which ones are important for understanding its meaning.

Analogy: Imagine you're in a noisy room full of conversations.

  • You hear everything (all tokens)
  • But you pay attention to specific voices that are relevant to what you're listening for
  • The attention mechanism assigns a "volume dial" to each voice — turn up the relevant ones, turn down the noise

The three key concepts:

  • Query (Q): "What am I looking for?" — the current word asking for context
  • Key (K): "What do I contain?" — each other word advertising its content
  • Value (V): "Here's my information" — the actual content to pull from matching words

Think of it like a library:

  • Query = your search term
  • Key = the title/index of each book
  • Value = the actual content of the book
  • Attention score = how well each book's title matches your search
  • Output = weighted blend of the most relevant books

This is THE core innovation behind GPT, BERT, Claude, and every modern language model. If you understand attention, you understand transformers.

# Exercise 4.6: Basic Attention Mechanism
# Interview scenario: "Explain and implement the attention mechanism
# used in transformers."

import numpy as np

def scaled_dot_product_attention(Q, K, V):
    """Scaled dot-product attention — the core of transformers.
    
    Q: queries  (seq_len, d_k)
    K: keys     (seq_len, d_k)
    V: values   (seq_len, d_v)
    
    Returns: attention output and weights
    """
    d_k = K.shape[-1]
    
    # Step 1: Compute attention scores
    scores = Q @ K.T / np.sqrt(d_k)  # scale by sqrt(d_k)
    
    # Step 2: Softmax to get attention weights (rows sum to 1)
    exp_scores = np.exp(scores - np.max(scores, axis=-1, keepdims=True))
    attention_weights = exp_scores / exp_scores.sum(axis=-1, keepdims=True)
    
    # Step 3: Weighted sum of values
    output = attention_weights @ V
    
    return output, attention_weights

5.2 Document Chunking — Breaking Documents Into Bite-Sized Pieces¶

The problem: An LLM has a limited context window (e.g., 8K tokens). Your company's policy manual is 200 pages. You can't feed the whole thing in. You also can't search over the whole thing — how would you know which part is relevant?

Solution: Split the document into chunks — small, overlapping pieces — and search over those.

Analogy: Think of a textbook. You don't read the entire book to answer one question. You check the index, find the relevant pages, and read those. Chunking creates the "pages" that your search system indexes.

Why overlapping? Imagine a sentence that spans two chunks. Without overlap, you'd split it in half and lose the meaning. Overlap ensures every sentence appears in full in at least one chunk.

Key decisions:

  • Chunk size: 400-500 characters is a good starting point
  • Overlap: 50-100 characters prevents information loss at boundaries
  • Sentence-aware splitting: Break at sentence boundaries, not mid-word
In [ ]:
# Exercise 5.2: Document Chunking
# Interview scenario: "Implement a text chunker with configurable size and overlap."

def chunk_text(text: str, chunk_size: int = 500, overlap: int = 100) -> list[dict]:
    """Split text into overlapping chunks.
    
    Args:
        text: The document to chunk
        chunk_size: Target characters per chunk
        overlap: Characters of overlap between consecutive chunks
    
    Returns:
        List of dicts with 'text', 'start', 'end', 'chunk_id'
    """
    chunks = []
    start = 0
    chunk_id = 0
    
    while start < len(text):
        end = start + chunk_size
        
        # Try to break at sentence boundary
        if end < len(text):
            # Look for period, question mark, or newline near the end
            for boundary in ['. ', '? ', '! ', '\n']:
                last_boundary = text[start:end].rfind(boundary)
                if last_boundary > chunk_size * 0.5:  # only if past halfway
                    end = start + last_boundary + len(boundary)
                    break
        
        chunk_text_str = text[start:end].strip()
        if chunk_text_str:
            chunks.append({
                'text': chunk_text_str,
                'start': start,
                'end': end,
                'chunk_id': chunk_id,
            })
            chunk_id += 1
        
        start = end - overlap  # overlap with previous chunk
        if start >= len(text):
            break
    
    return chunks

5.3 Vector Search with FAISS — Finding Needles in Haystacks¶

The problem: You have 10,000 document chunks, each converted to a 384-dimension vector. A user asks a question (also converted to a vector). How do you find the most relevant chunks — fast?

FAISS (Facebook AI Similarity Search) is a library that finds the most similar vectors in milliseconds, even with millions of vectors.

Analogy: Imagine a library with 10,000 books, but instead of being organized by author or title, each book has a GPS coordinate based on its content. Books about similar topics are physically close together. To find books about "machine learning", you just look at the GPS coordinates near your query and grab the closest ones.

That's exactly what vector search does — except the "GPS coordinates" have 384 dimensions instead of 2.

Why FAISS? It's free, runs locally, and handles millions of vectors. In production, managed services like Pinecone or Weaviate add scaling and persistence on top of the same idea.

5.4 The RAG Pipeline — Putting It All Together¶

RAG = Retrieval-Augmented Generation. It's the most common architecture in production AI systems today.

The flow in plain English:

  1. Load your documents
  2. Chunk them into small pieces
  3. Embed each chunk into a vector
  4. Store vectors in a search index (FAISS)
  5. When a user asks a question:
    • Embed the question
    • Search for the most similar chunks
    • Feed those chunks + the question to an LLM
    • LLM generates an answer grounded in your documents

Analogy: You're an assistant in a law firm.

  • A lawyer asks you a question about a case
  • You don't know the answer from memory (you're not trained on this firm's cases)
  • So you go to the filing cabinet, pull the 3 most relevant case files
  • You read them and write a summary that answers the lawyer's question
  • That's RAG. The filing cabinet is the vector database. You are the LLM.

Why not just ask the LLM directly? LLMs don't know your private data. They hallucinate when they don't know something. RAG gives them the right documents so they can answer accurately.

import numpy as np
import faiss

class SimpleRAG:
    """Minimal RAG pipeline: load → chunk → embed → retrieve → generate."""
    
    def __init__(self, dim: int = 384, chunk_size: int = 300, overlap: int = 50):
        self.dim = dim
        self.chunk_size = chunk_size
        self.overlap = overlap
        self.index = faiss.IndexFlatIP(dim)
        self.chunks = []
    
    def _chunk(self, text: str) -> list[str]:
        """Split text into overlapping chunks."""
        result = []
        start = 0
        while start < len(text):
            end = min(start + self.chunk_size, len(text))
            result.append(text[start:end].strip())
            start = end - self.overlap
            if start >= len(text):
                break
        return [c for c in result if c]
    
    def _embed(self, texts: list[str]) -> np.ndarray:
        """Mock embedding function (replace with real model in production)."""
        vecs = np.array([
            np.random.RandomState(hash(t) % 2**31).randn(self.dim).astype(np.float32)
            for t in texts
        ])
        faiss.normalize_L2(vecs)
        return vecs
    
    def _mock_llm(self, prompt: str) -> str:
        """Mock LLM response (replace with API call in production)."""
        # Extract context and question from prompt
        lines = prompt.split('\n')
        return f"[Mock LLM Response] Based on the {len(lines)} lines of context provided, " \
               f"here is a synthesized answer to your question."
    
    def ingest(self, documents: list[str]) -> int:
        """Ingest documents: chunk → embed → index."""
        for doc in documents:
            doc_chunks = self._chunk(doc)
            self.chunks.extend(doc_chunks)
        
        embeddings = self._embed(self.chunks)
        self.index.add(embeddings)
        return len(self.chunks)
    
    def query(self, question: str, top_k: int = 3) -> dict:
        """Full RAG query: embed question → retrieve → generate."""
        # Retrieve
        q_vec = self._embed([question])
        scores, indices = self.index.search(q_vec, k=top_k)
        
        retrieved = [
            {'text': self.chunks[idx], 'score': float(scores[0][i])}
            for i, idx in enumerate(indices[0])
        ]
        
        # Generate
        context = '\n\n'.join([r['text'] for r in retrieved])
        prompt = f"""Answer the question using ONLY the context below.

Context:
{context}

Question: {question}

Answer:"""
        answer = self._mock_llm(prompt)
        
        return {
            'answer': answer,
            'retrieved_chunks': retrieved,
            'prompt_length': len(prompt),
        }

5.6 Re-Ranking — When the First Search Isn't Good Enough¶

The problem: Your vector search returns 10 results. Result #3 is actually the most relevant, but it ranked lower because vector search only measures semantic similarity. It missed that the query and document share specific keywords.

Re-ranking takes the initial search results and re-orders them using additional signals.

Analogy: You Google something and get 10 results. Google's first-pass algorithm found them. Then a second algorithm re-ranks them based on your click history, location, and the page's authority. The final order is different from the first pass.

Common re-ranking strategies:

  • Keyword overlap — does the document contain the exact words from the query? (catches what semantic search misses)
  • Cross-encoder models — a neural network that looks at the query AND document together (more accurate but slower)
  • Hybrid scoring — combine semantic score + keyword score with weights

Why not just do better search? No single search method is perfect. Semantic search understands meaning but misses keywords. Keyword search finds exact matches but misses synonyms. Combining them gets the best of both worlds.

# Exercise 5.6: Re-Ranking Retrieved Documents
# Interview scenario: "Your RAG retrieves relevant docs but the order is wrong.
# Implement a re-ranker."

import numpy as np

def keyword_overlap_score(query: str, text: str) -> float:
    """Simple keyword overlap between query and document."""
    query_words = set(query.lower().split())
    text_words = set(text.lower().split())
    if not query_words:
        return 0.0
    return len(query_words & text_words) / len(query_words)

def rerank(
    query: str,
    retrieved: list[dict],
    semantic_weight: float = 0.6,
    keyword_weight: float = 0.4,
) -> list[dict]:
    """Re-rank retrieved documents using hybrid scoring.
    
    Combines:
    - Semantic similarity (from initial retrieval)
    - Keyword overlap (lexical matching)
    """
    for doc in retrieved:
        kw_score = keyword_overlap_score(query, doc['text'])
        doc['keyword_score'] = kw_score
        doc['combined_score'] = (
            semantic_weight * doc['semantic_score'] +
            keyword_weight * kw_score
        )
    
    return sorted(retrieved, key=lambda x: x['combined_score'], reverse=True)

5.7 Tool-Calling Agent — AI That Takes Actions¶

The problem: A chatbot can answer questions, but what if the user says "What's the status of my order?" The answer isn't in any document — you need to call an API to look it up.

An agent is an LLM that can decide which tool to use to answer a question, then use it.

Analogy: Think of a receptionist.

  • Someone calls and asks about their order → receptionist checks the order system (tool 1)
  • Someone asks about the weather → receptionist checks the weather app (tool 2)
  • Someone asks to add two numbers → receptionist uses a calculator (tool 3)
  • Someone asks a general question → receptionist answers from knowledge

The receptionist doesn't know everything, but knows which tool to reach for. That's an agent.

How it works:

  1. Register tools — define what each tool does (name, description, function)
  2. Route — match the user's query to the right tool
  3. Execute — call the tool and get the result
  4. Respond — format the result for the user

In production: LangGraph, CrewAI, and OpenAI function calling all implement this pattern with the LLM itself deciding which tool to call.

class SimpleAgent:
    """Agent that routes user queries to appropriate tools."""

        self.conversation_history: list[dict] = []
    
    def register_tool(self, tool: Tool):
        self.tools[tool.name] = tool
    
    def _route(self, query: str) -> str | None:
        """Simple keyword-based routing (in production: LLM decides)."""
        query_lower = query.lower()
        # Score each tool by keyword overlap with its description
        best_tool = None
        best_score = 0
        for name, tool in self.tools.items():
            desc_words = set(tool.description.lower().split())
            query_words = set(query_lower.split())
            score = len(desc_words & query_words)
            if score > best_score:
                best_score = score
                best_tool = name
        return best_tool if best_score > 0 else None
    
    def run(self, query: str) -> dict:
        """Process a user query: route → execute tool → return result."""
        self.conversation_history.append({'role': 'user', 'content': query})
        
        tool_name = self._route(query)
        
        if tool_name and tool_name in self.tools:
            tool = self.tools[tool_name]
            result = tool.handler(query)
            response = {
                'tool_used': tool_name,
                'result': result,
                'status': 'success',
            }
        else:
            response = {
                'tool_used': None,
                'result': 'I can help with order lookups, weather, and calculations.',
                'status': 'no_tool_matched',
            }
        
        self.conversation_history.append({'role': 'assistant', 'content': str(response)})

6.1 ML Fundamentals¶

Q: What is the bias-variance tradeoff?

Bias = error from wrong assumptions (underfitting). The model is too simple to capture the pattern. Variance = error from sensitivity to training data (overfitting). The model memorizes noise. Tradeoff: Increasing model complexity reduces bias but increases variance. The goal is the sweet spot.

Go deeper: "How do you detect if your model is high-bias vs high-variance?" → Compare train vs validation loss. High bias = both high. High variance = train low, val high.

Q: What is overfitting and how do you prevent it?

Model performs well on training data but poorly on unseen data. It memorized the noise.

Technique How It Helps
More data Harder to memorize with more examples
Regularization (L1/L2) Penalizes large weights
Dropout Randomly disables neurons during training
Early stopping Stop training when validation loss stops improving
Cross-validation Evaluate on multiple train/val splits
Data augmentation Create variations of training examples

Q: Explain cross-validation.

Split data into K folds. Train on K-1 folds, validate on the remaining fold. Repeat K times. Final score = average across all folds. Gives a more robust estimate than a single train/test split.

Go deeper: "When would you NOT use K-fold?" → Time-series data (use temporal splits), very large datasets (single split is fine), or when training is too expensive.

Q: When would you use L1 vs L2 regularization?

L1 (Lasso): Drives some weights to exactly zero → feature selection. Use when you suspect many features are irrelevant. L2 (Ridge): Shrinks all weights toward zero but doesn't eliminate them. Use when all features are potentially useful. Elastic Net: Combination of both. Good default choice.

6.2 Deep Learning & Transformers¶

Q: How does the transformer architecture work?

Core mechanism: self-attention — each token looks at every other token to understand context.

Component Purpose
Input Embedding Convert tokens to vectors
Positional Encoding Add position info (attention has no inherent order)
Multi-Head Attention Multiple attention patterns in parallel
Feed-Forward Network Process each position independently
Layer Norm + Residual Stabilize training
Stack N layers Depth = more complex representations

Go deeper: "Why scale by sqrt(d_k)?" → Without scaling, dot products grow large with dimension, making softmax saturate (all attention on one token).

Q: What's the difference between encoder and decoder architectures?

Architecture Sees Used For Example
Encoder (BERT) Full sequence (bidirectional) Classification, embeddings, NER "Is this spam?"
Decoder (GPT) Only past tokens (causal/autoregressive) Text generation "Complete this sentence..."
Encoder-Decoder (T5) Encoder sees full input, decoder generates output Translation, summarization "Translate to French"

Q: What is positional encoding and why is it needed?

Attention is permutation-invariant — it doesn't know word order. Positional encoding adds position information. Original paper uses sine/cosine functions. Modern models (GPT, Llama) use learned or rotary embeddings (RoPE).

Go deeper: "What's RoPE?" → Rotary Position Embedding. Encodes position as a rotation in embedding space. Better extrapolation to longer sequences than absolute position embeddings.

6.3 LLM-Specific Questions¶

Q: What is temperature in LLM generation?

Temperature controls randomness. Applied to logits before softmax.

Temperature Effect Use Case
0.0 Deterministic (greedy) Factual Q&A, structured output
0.3-0.7 Balanced General conversation
1.0+ More random/creative Creative writing, brainstorming

Formula: softmax(logits / temperature)

Q: How does tokenization work?

Text → sequence of integer token IDs. Common methods:

  • BPE (GPT): Byte Pair Encoding — merges frequent byte pairs iteratively
  • WordPiece (BERT): Similar but uses likelihood-based merging
  • SentencePiece (Llama, T5): Language-agnostic, treats text as raw bytes

Why it matters: Token limits, cost (billed per token), and edge cases ("tokenization" may split into 3+ tokens).

Q: What is RLHF and why does it matter?

Reinforcement Learning from Human Feedback. Three stages:

  1. Pre-train on internet text (predict next token)
  2. SFT (Supervised Fine-Tuning) on human-written examples
  3. RLHF — train a reward model on human preferences, then optimize the LLM against it

RLHF is what makes ChatGPT helpful vs. just a text predictor. Alternative: DPO (Direct Preference Optimization) — skips the reward model.

Q: What is LoRA and why is it used for fine-tuning?

LoRA (Low-Rank Adaptation): Instead of updating all model weights (billions), inject small trainable matrices into each layer. Trains ~0.1-1% of parameters. Same quality as full fine-tuning for most tasks.

Method Parameters Trained Memory Quality
Full fine-tuning 100% Very high Best
LoRA 0.1-1% Low Near-best
QLoRA 0.1-1% (4-bit base) Very low Good
Prompt tuning < 0.01% Minimal Moderate

Q: Why do LLMs hallucinate and how do you mitigate it?

LLMs generate plausible text, not factual text. They have no notion of truth — only patterns.

Mitigation How It Works
RAG Ground generation in retrieved documents
Lower temperature Reduce randomness
Prompt engineering "Only answer based on the context provided"
Citation Ask model to cite sources (easier to verify)
Fine-tuning Train on domain-specific data
Guardrails Post-generation validation and filtering

import numpy as np

def softmax_with_temperature(logits, temperature=1.0):
    """Apply temperature scaling and softmax."""
    scaled = logits / max(temperature, 1e-8)
    exp = np.exp(scaled - np.max(scaled))
    return exp / exp.sum()

# Simulate: model's raw output (logits) for next token
tokens = ['Paris', 'London', 'Berlin', 'Tokyo', 'Madrid']
logits = np.array([3.5, 2.1, 1.8, 1.5, 0.8])  # model thinks 'Paris' is most likely

print('=== Temperature Effect on Token Probabilities ===')
print(f'\nLogits: {dict(zip(tokens, logits))}\n')

for temp in [0.1, 0.5, 1.0, 2.0]:
    probs = softmax_with_temperature(logits, temp)
    print(f'Temperature = {temp}:')
    for token, p in zip(tokens, probs):
        bar = '#' * int(p * 40)
        print(f'  {token:>8}: {p:.3f} {bar}')
    print()

print('--- Low temp → confident (deterministic). High temp → spread out (creative). ---')

What evaluators look for in your answer:

Getting Real Experience Without a Job¶

You don't need to be employed as an AI Engineer to have AI engineering experience.

# Strategy What You Get
1 Turn projects into products Deploy your RAG chatbot. Get 10 users. Fix what breaks. Now it's a production system, not a homework assignment.
2 Solve problems for small organizations Local nonprofits, small businesses, student orgs — they need AI tools but can't afford engineers. Real users = real resume bullets.
3 Build AI tools for your own workflow Summarize meeting notes, organize your job search, review your own code. If you use it daily, you'll discover real production issues.
4 Contribute to engineering communities Open-source AI projects, Stack Overflow, technical blog posts. Public contributions are visible proof of competence.
5 Build systems that handle imperfect data Real data is messy — missing fields, inconsistent formats, duplicates. Build pipelines that handle this gracefully. This is what separates junior from mid-level.

The Mindset Shift¶

Stop asking "does this run?"

Start asking "does this system survive real users?"

That single question changes everything about how you build.

Phase What You Do What It Proves
1. Learn Complete exercises, understand the concepts Foundation
2. Build Create a working project from scratch Technical ability
3. Deploy Put it on a server with a URL anyone can visit Production thinking
4. Get users Share it, get feedback, watch it break Real-world exposure
5. Fix problems Handle edge cases, add monitoring, improve UX Engineering maturity

Most people stop at Phase 2. The ones who get hired complete Phase 5.
